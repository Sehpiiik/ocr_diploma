# OCR Datasets — Унифицированные аннотации на уровне строк

Python проект, который стандартизирует OCR датасеты с гетерогенными форматами аннотаций в единую JSON схему на уровне строк. Каждый входной датасет использует своё собственное представление (пров слово `.pkl`, пословный `.json`, построчный `.json`); этот проект преобразует их все в одинаковую структуру, чтобы последующие конвейеры обучения / оценки могли обрабатывать их единообразно.

## Унифицированная выходная схема

Каждый сгенерированный файл выглядит так:

```json
{
  "image": "ddi_10_0.png",
  "domain": "ddi",
  "width": 2480,
  "height": 3508,
  "bbox_format": "xywh",
  "objects": [
    {
      "text": "Admission by consent of the instructor.",
      "bbox": [200, 200, 2030, 57],
      "type": "line"
    }
  ]
}
```

* ``domain`` - исходный датасет: ``"ddi"``, ``"rdtag"`` или ``"smartdoc"``.
* ``image`` - базовое имя фактического файла, находящегося рядом с аннотацией внутри ``output/<dataset>/`` — каждый ``<name>.json`` имеет соседний файл ``<name>.png/.jpg``. Смотрите [Структура выходных данных](#структура-выходных-данных) для правил именования.
* ``objects[*].type`` всегда ``"line"`` - пословные входные данные объединяются в строки, построчные входные данные передаются как есть.

Используются два значения ``bbox_format``:

* ``"xywh"`` - выровненный по осям прямоугольник ``[x, y, w, h]`` в пикселях. Используется для **DDI100**, исходные данные которого представляют собой отрендеренную (идеально выровненную) страницу.
* ``"polygon"`` - 4-точечный четырёхугольник ``[[x, y], [x, y], [x, y], [x, y]]`` в порядке ``[TL, TR, BR, BL]`` в пикселях. Используется для **RDTAG** (выровненные по осям прямоугольники, выраженные как четырёхугольники для единообразия схемы) и **SmartDoc** (настоящие перспективные четырёхугольники с фотографированных страниц).

## Структура выходных данных

Каждый запуск преобразования создаёт самодостаточную папку для каждого датасета, разделённую на поддиректории ``images/`` и ``annotations/``. Исходные изображения жёстко связываются под их конечными именами файлов (обычные копии используются как запасной вариант, когда жёсткие ссылки не поддерживаются), поэтому исходные данные не изменяются и в типичном случае нет дублирования на диске.

```
output/
├── ddi100/
│   ├── images/
│   │   └── ddi_<doc>_<page>.png        # например, ddi_10_0.png
│   └── annotations/
│       └── ddi_<doc>_<page>.json       # ссылается на .png по базовому имени
├── rdtag/
│   ├── images/
│   │   └── <original_name>.jpg         # например, 5_Book7_120_in_ni_y_4.jpg (оставляется как есть)
│   └── annotations/
│       └── <original_name>.json
└── smartdoc/
    ├── images/
    │   └── smartdoc_<id>.jpg           # например, smartdoc_00001.jpg
    └── annotations/
        └── smartdoc_<id>.json
```

Правила именования:

* **DDI100**: изображение и аннотация называются ``ddi_<папка_документа>_<страница>`` (например, ``ddi_10_0``). Исходные PNG находятся внутри директорий ``orig_texts/`` для каждого документа с числовыми именами без расширения; конвертер переименовывает их, создавая жёсткие ссылки в ``output/ddi100/images/``.
* **RDTAG**: исходное имя файла RDTAG сохраняется буквально — как для изображения (``5_Book7_120_in_ni_y_4.jpg``), так и для соответствующей аннотации ``.json``.
* **SmartDoc**: исходные изображения используют числовые имена без расширения (``00001.jpg``); конвертер переименовывает их в ``smartdoc_00001.jpg`` и создаёт рядом ``smartdoc_00001.json``.

Поле ``image`` в аннотации всегда является базовым именем - чтобы загрузить изображение, объедините его с соседней директорией ``images/`` файла аннотации.

## Структура проекта

```
ocr_datasets/
├── convert.py                       # Точка входа CLI (оркестрирует конвертеры)
├── converters/
│   ├── __init__.py
│   ├── ddi100_converter.py          # DDI100 .pkl  → унифицированный JSON

│   └── smartdoc_converter.py        # SmartDoc2015 .json → унифицированный JSON

│   ├── images_rotation.py           # Поворот + обрезка краёв для папки изображений

```

## Датасеты

### DDI100 v1.3

Исходная структура:

```
DDI100_v1.3/
└── <doc_id>/
    ├── orig_texts/<page>.png        # отрендеренные страницы документов
    └── orig_boxes/<page>.pkl        # пословные аннотации (pickled list of dicts)
```

Каждый `.pkl` представляет собой `list[dict]` с полями `text`, `box` и `chars`. `box` — это 4-точечный массив `numpy.int32` в порядке `(row, col)` — т.е. каждая строка это `(y, x)`. Конвертер извлекает выровненные по осям границы `xywh` из четырёх угловых точек, затем группирует слова в строки по вертикальной близости.

### RDTAG-1.0

Распространяется со всем содержимым, смешанным внутри каждого поднабора:

```
RDTAG-1.0/
└── Training_Set<N>/
    ├── <name>.jpg
    ├── <name>.json     
    └── <name>.txt      
```

Конвертер ожидает реорганизованную структуру, созданную `datasets_utils/reorganize_rdtag.py`:

```
RDTAG-1.0/
├── images/
│   └── Training_Set<N>/<name>.jpg
└── orig_annotations/
    └── Training_Set<N>/
        ├── <name>.json
        └── <name>.txt
```

Запустите шаг реорганизации перед вызовом конвертера (см. [Утилиты](#утилиты)). Если ожидаемые директории `images/` и `orig_annotations/` отсутствуют, `convert_rdtag` вызовет `FileNotFoundError`, указывая на скрипт.

### SmartDoc2015_Challenge_2

```
SmartDoc2015_Challenge_2/
├── images/<id>.jpg
├── ocr_boxes/<id>.json              
└── input_sample_groundtruth/<id>.txt
```

Каждый `ocr_boxes/<id>.json` выглядит так:

```json
{
  "image": "00040.jpg",
  "detections": [
    {"bbox": {"polygon": [[311, 338], [1456, 323], [1457, 370], [311, 386]]},
     "text": "...", "confidence": 0.94}
  ]
}
```

Каждый полигон представляет собой 4-точечный четырёхугольник в порядке `[TL, TR, BR, BL]` - непрямоугольная форма отражает перспективный наклон сфотографированной страницы.

Аннотации в основном построчные, но OCR движок иногда выдаёт рамки отдельных слов (например, номер раздела рядом с его заголовком). Конвертер группирует обнаружения в строки, используя вертикальный IOU + предикат отсутствия горизонтального перекрытия, затем выдаёт каждую строку как 4-точечный полигон. При слиянии в одной строке выходной полигон формируется из левого края самого левого элемента и правого края самого правого элемента, чтобы сохранить исходный наклон.

## Использование

### Конвертировать все датасеты

```bash
python convert.py                  # записывает output/{ddi100,rdtag,smartdoc}/
```

### Конвертировать один датасет

```bash
python convert.py --dataset ddi100
python convert.py --dataset rdtag
python convert.py --dataset smartdoc
```

### Пользовательская выходная директория

```bash
python convert.py --output /path/to/out
```

## Утилиты

### `datasets_utils/reorganize_rdtag.py`

Разделяет RDTAG-1.0 на структуру `images/` + `orig_annotations/`, сохраняя структуру поддиректорий `Training_Set<N>` внутри каждой. Таким образом избегаются коллизии имён файлов, и группировка по обучающим сплитам сохраняется.

```bash
# Просмотр без изменений файловой системы
python datasets_utils/reorganize_rdtag.py -d datasets/RDTAG-1.0 --dry-run

# Перемещение файлов
python datasets_utils/reorganize_rdtag.py -d datasets/RDTAG-1.0

# Копирование вместо перемещения (сохраняет исходную структуру)
python datasets_utils/reorganize_rdtag.py -d datasets/RDTAG-1.0 --copy
```

Скрипт идемпотентен: если директории `Training_Set*/` уже удалены, он выводит уведомление и завершает работу без ошибок.

### `datasets_utils/images_rotation.py`

Поворачивает каждое изображение в папке на фиксированный угол и опционально обрезает постоянный отступ с каждого края. Полезно для нормализации сканированных датасетов, где все страницы были захвачены повёрнутыми как, например, в SmartDoc 2015.

```bash
python datasets_utils/images_rotation.py \
    --input  datasets/RDTAG-1.0/images/Training_Set1 \
    --output rotated/Training_Set1 \
    --format jpg \
    --angle 270 \
    --crop 0
```

## Примечания по реализации

* **Группировка слов в строки.** Для DDI100 и RDTAG слова группируются в одну строку, когда абсолютная разница по Y между их верхними краями составляет менее 50% от высоты опорного слова. Внутри строки слова сортируются по X и объединяются с одним пробелом; bbox строки представляет собой выровненное по осям объединение bbox слов
* **Многоколоночные документы (DDI100).** После Y-группировки каждая строка снова разделяется везде, где между последовательными словами есть горизонтальный разрыв более 1.5 × их высоты. Это предотвращает объединение слов из соседних колонок (например, ячеек таблицы, параллельных абзацев) в одну строку
* **Поиск изображений.** Ширина/высота считываются непосредственно из файла изображения с помощью Pillow. Если изображение отсутствует, `width` и `height` записываются как `0`, и преобразование всё равно завершается успешно
* **Размещение изображений.** Исходные изображения жёстко связываются в ``output/<dataset>/`` под их унифицированными именами; в файловых системах, не поддерживающих жёсткие ссылки, конвертер переключается на обычное копирование. Существующие файлы назначения остаются нетронутыми
* **Идемпотентный вывод.** Каждый запуск перезаписывает JSON файлы назначения; частичные запуски безопасно возобновлять
